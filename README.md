# О проекте

mini_nginx это асинхронный HTTP reverse proxy на `asyncio`. Учебный проект:
принимает клиентские соединения, парсит HTTP-запрос, проксирует его на один из
апстримов по схеме round-robin и возвращает ответ клиенту.

## Возможности

- TCP-сервер на `asyncio.start_server` с логированием входящих соединений
- Парсер HTTP-запроса: стартовая строка (метод, путь, версия) + заголовки;
  тело передаётся как raw-стрим
- Двунаправленное проксирование с backpressure через `drain()`
- Round-robin балансировка по нескольким апстримам
- Таймауты на connect / read / write и общий total
- Лимиты одновременных соединений через `asyncio.Semaphore`
  (отдельно на клиентов и на коннекты к апстриму)
- Логирование: апстрим-хост, метод/путь, статус ответа

## Структура проекта

**`server.py`** — Поднимает TCP-сервер через `asyncio.start_server`,
принимает клиентские соединения и на каждое обрабатывает запрос: парсит голову
HTTP-запроса, выбирает апстрим, проксирует данные в обе стороны (запрос → апстрим,
ответ → клиент). Здесь же — таймауты, лимит клиентских соединений и логирование.

**`upstream_pool.py`** — класс `UpstreamPool`, отвечающий за апстримы. Хранит их
список, раздаёт по кругу (round-robin) через `next_upstream()` и ограничивает
число одновременных соединений к апстримам через семафор (`limit()`).

**`echo_app.py`** — тестовый апстрим на FastAPI, не часть самого прокси, а бэкенд
для проверки. Отражает запросы обратно: `GET /` отвечает `{"message": "ok"}`,
`POST /echo` возвращает присланное тело. Каждый ответ содержит поле `upstream` с
именем сервера (из переменной `UPSTREAM_NAME`) — по нему видно, какой бэкенд
обслужил запрос, что нужно для проверки round-robin под нагрузкой. Запускается
в нескольких экземплярах на разных портах.

**`config.py`** — конфигурация проекта в одном месте: таймауты (connect/read/write/total),
лимиты соединений (клиентские и к апстримам)

**`utils/http_parser.py`** — разбор HTTP-запроса. Содержит датакласс `RequestHead`
(метод, путь, версия, заголовки) и функцию `parse_request_head`, которая парсит
голову запроса из сырых байтов.

## Установка

```bash
poetry install
```
## Конфигурация

Все параметры задаются в файле `config.yaml` в корне проекта:

```yaml
listen: "127.0.0.1:8080"          # адрес и порт, на которых слушает прокси
upstreams:                         # апстримы для round-robin
  - host: "127.0.0.1"
    port: 9001
  - host: "127.0.0.1"
    port: 9002
  - host: "127.0.0.1"
    port: 9003
timeouts:                          # таймауты в миллисекундах
  connect_ms: 1000                 # на установку соединения с апстримом
  read_ms: 15000                   # на отдельный read
  write_ms: 15000                  # на drain
  total_ms: 30000                  # на всю пересылку целиком
limits:
  max_client_conns: 1000           # лимит клиентских соединений
  max_conns_per_upstream: 100      # лимит коннектов к апстриму
logging:
  level: "info"                    # уровень логирования
```

Чтобы изменить порт, список апстримов или таймауты — правится `config.yaml`,
код менять не нужно. Таймауты в конфиге задаются в миллисекундах и при загрузке
переводятся в секунды.

## Запуск

### 1. Поднять апстримы (echo-серверы на FastAPI)

В отдельных терминалах или в фоне — апстримы на портах, указанных в `config.yaml`.
Переменная `UPSTREAM_NAME` задаёт имя апстрима: оно возвращается в каждом
ответе (поле `upstream`) и позволяет видеть, какой бэкенд обслужил запрос —
это нужно для проверки round-robin под нагрузкой.

```bash
UPSTREAM_NAME=up-9001 poetry run uvicorn mini_nginx.echo_app:app --host 127.0.0.1 --port 9001 --workers 1 &
UPSTREAM_NAME=up-9002 poetry run uvicorn mini_nginx.echo_app:app --host 127.0.0.1 --port 9002 --workers 1 &
UPSTREAM_NAME=up-9003 poetry run uvicorn mini_nginx.echo_app:app --host 127.0.0.1 --port 9003 --workers 1 &
```

Порты апстримов должны совпадать со списком `upstreams` в `config.yaml`.

Остановить все апстримы:

```bash
pkill -f "uvicorn mini_nginx.echo_app"
```

### 2. Запустить прокси

```bash
poetry run python -m mini_nginx.server
```

Прокси слушает адрес из `config.yaml` (по умолчанию `127.0.0.1:8080`).

## Проверка

```bash
curl -v http://127.0.0.1:8080/
curl -v -X POST http://127.0.0.1:8080/echo -d 'hello world'
```

Несколько запросов подряд — видно, как round-robin перебирает апстримы:

```bash
for i in 1 2 3 4; do curl -s http://127.0.0.1:8080/ ; echo ; done
```

В ответах чередуются `up-9001 → up-9002 → up-9003 → up-9001`.

## Нагрузочное тестирование (k6)

Нагрузочный тест прогоняется через [k6](https://grafana.com/docs/k6/latest/set-up/install-k6/).
Сценарий `load_test.js` шлёт `GET /` и `POST /echo`, проверяет статусы и
собирает распределение ответов по апстримам.

Установка k6 (macOS):

```bash
brew install k6
```

Запуск (апстримы и прокси должны быть подняты):

```bash
k6 run load_test.js
```

Сценарий: 20 виртуальных пользователей в течение 30 секунд. Пороги:
p95 latency < 500ms, p99 < 1000ms, доля ошибок < 1%.

### Результаты прогона

| Метрика | Значение |
|---------|----------|
| RPS | ~22 600 req/s |
| latency p95 | 1.31 ms |
| latency p99 | 1.62 ms |
| ошибки (`http_req_failed`) | 0.00% |
| timeouts | 0 |

Распределение по апстримам (round-robin) — нагрузка раскидана равномерно:

| Апстрим | Запросов |
|---------|----------|
